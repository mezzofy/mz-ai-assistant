## Summary

This release fixes a root-cause bug where the History tab showed no Task ID badge for
**failed** tasks and **in-progress** tasks. Previously, COMPLETED tasks showed the badge
correctly, but any task whose Celery worker failed or was still running showed a blank
session card with no task context.

---

## Bug Fixes

### BUG-005 — History tab: Task ID badge absent for failed and in-progress tasks

**Root cause:** `agent_tasks.session_id` was NULL at task-creation time for new conversations.

When a long-running task was triggered from a new chat (no existing `session_id`):

1. The server inserted into `agent_tasks` with `session_id = NULL` (because `body.session_id`
   was `None` for a brand-new conversation).
2. The 202 response did **not** include `session_id`.
3. The mobile store set `newActiveTask.session_id = null` (from `response.session_id ?? null`).
4. `HistoryScreen.tsx` grouping skipped any task where `t.session_id` was falsy → **no badge**.

For failed tasks: if the Celery worker failed before writing `session_id` back via
`_update_agent_task_session()`, the column stayed NULL permanently → badge never appeared.

For in-progress tasks: `session_id` was NULL during execution → badge absent on every refresh.

---

**Fix — Backend (`server/app/api/chat.py`):**

The long-running task path now calls `get_or_create_session()` **synchronously** (within the
same request, before the `agent_tasks` INSERT). The resolved session ID is:
- Stored in `agent_tasks.session_id` at INSERT time → **never NULL**
- Passed in `task_payload` to the Celery worker → worker reuses the existing session
- Returned in the 202 response body as `session_id`

The `_update_agent_task_session()` helper in `tasks.py` is now a no-op for new conversations
(the `if not session_id` guard is False because session_id is already populated), but the
helper is retained for safety.

**Fix — Mobile (`APP/src/stores/chatStore.ts`):**

The queued-response branch in `sendToServer()` now sets `sessionId: response.session_id || null`
in the Zustand store. This links the active chat screen to the queued session immediately,
enabling correct polling and eventual WS notification routing.
